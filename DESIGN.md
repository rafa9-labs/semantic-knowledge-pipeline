## 1. Product Vision

### What Is DevKnowledge?

An AI-powered learning platform where **ML/AI engineers** can quickly master the modern
data/AI tech stack through structured knowledge graphs, simple explanations, practical
code examples, and interactive exercises.

### The "Eat Your Own Dogfood" Principle

This tool teaches the **exact tech stack used to build it**. When a learner studies
Pydantic, they see the actual Pydantic models from this pipeline. When they learn about
embeddings, they see the real embedding code powering their search. The app is its own
textbook.

### Target User

- **Primary:** Junior-to-mid ML/AI engineers, computer engineering students
- **Goal:** Learn modern tools fast — not read 300-page manuals, but grasp key concepts,
  see how they connect, and practice with real code
- **Use cases:**
  - "I know Python basics but need to understand async programming for my ML pipeline"
  - "I need to learn FastAPI to deploy my model"
  - "How do vector databases actually work?"
  - "What's RAG and how do I build one?"

## 3. The 4 Content Layers

Every concept in the system has **4 layers** of content, each serving a different
learning need:

### Layer 1: 📘 THEORY — Formal Explanation
- **Source:** Scraped from official documentation (MDN, Python docs, FastAPI docs, etc.)
- **Purpose:** The authoritative, precise definition
- **Example:**
  > "The `async` keyword declares an asynchronous function. When called, it returns a
  > Promise. The function's body runs asynchronously via the event loop, allowing
  > non-blocking I/O operations."

### Layer 2: 📗 SIMPLE EXPLANATION — "Explain Like I'm 5"
- **Source:** Generated by our LLM (Gemma 4)
- **Purpose:** Plain-language analogy that builds intuition before rigor
- **Example:**
  > "Think of `async/await` like ordering at a restaurant. You place your order (call the
  > async function) and get a receipt (Promise). While the kitchen cooks (I/O operation),
  > you can chat with friends (other code runs). When the food's ready, the receipt
  > resolves and you get your meal (the return value)."

### Layer 3: 📙 EXAMPLES — Real Code Snippets
- **Source:** Scraped from docs + generated by LLM
- **Purpose:** Show the concept in practice with annotated code
- **Example:**
  > ```python
  > # Example: Basic async function in Python
  > import asyncio
  >
  > async def fetch_data(url: str) -> str:
  >     """Simulates fetching data from an API."""
  >     await asyncio.sleep(1)  # Non-blocking wait
  >     return f"Data from {url}"
  >
  > # Run it
  > result = asyncio.run(fetch_data("https://api.example.com"))
  > print(result)  # "Data from https://api.example.com"
  > ```

### Layer 4: 📕 EXERCISES — Practice Problems
- **Source:** Generated by our LLM with test cases
- **Purpose:** Hands-on practice to solidify understanding
- **Example:**
  > **Title:** Build a Concurrent URL Fetcher
  >
  > **Description:** Write an async function that fetches 3 URLs concurrently using
  > `asyncio.gather()`. It should return a list of results in the same order as the URLs.
  >
  > **Starter Code:**
  > ```python
  > import asyncio
  >
  > async def fetch_many(urls: list[str]) -> list[str]:
  >     # TODO: Your code here
  >     pass
  > ```
  >
  > **Hints:**
  > 1. Create a list of coroutines: `[fetch_one(url) for url in urls]`
  > 2. Pass them to `asyncio.gather()`
  >
  > **Solution:** (hidden until requested)
  > ```python
  > async def fetch_many(urls: list[str]) -> list[str]:
  >     tasks = [fetch_one(url) for url in urls]
  >     return await asyncio.gather(*tasks)
  > ```

**Relationship types:**
- `requires` — You must know A before B (async requires understanding of functions)
- `enables` — Knowing A unlocks B (understanding Promises enables async/await)

- `part_of` — A is a component of B (routing is_part_of FastAPI)
- `related_to` — General connection (Promises related_to callbacks)
- `contrasts_with` — A and B are alternatives/opposing (sync contrasts_with async)
- `built_on` — A is built on top of B (FastAPI built_on Starlette)

## 7. Data Enrichment Pipeline

After scraping raw articles, several enrichment steps produce the 4 content layers:

```
Raw Article (scraped)
     │
     ├──→ Parse into Sections           → source_sections table
     │
     ├──→ Extract Concepts (LLM)        → concepts table
     │
     ├──→ Generate ELI5 (LLM)           → concepts.simple_explanation
     │
     ├──→ Extract Relationships (LLM)   → concept_relationships table
     │
     ├──→ Extract Examples (LLM)        → examples table (source=scraped)
     │
     ├──→ Generate Examples (LLM)       → examples table (source=generated)
     │
     ├──→ Generate Exercises (LLM)      → exercises table
     │
     └──→ Embed Concept (Ollama)         → Weaviate (for semantic search)
```

## 10. Key Design Decisions

### 10.1 Concepts vs Triples — Why Both?

- **Triples** (existing): Raw (subject, predicate, object) facts — great for extraction,
  but subjects/objects are free-text strings, not linked to anything
- **Concepts** (new): First-class entities with IDs, names, categories, explanations
- **Concept Relationships** (new): Typed edges between concept IDs — the actual graph

**Migration path:** Existing triples are used to generate concept relationships.
The triple "async function → returns → Promise" becomes a relationship between
the "async function" concept and the "Promise" concept with type `enables`.

### 10.2 LLM-Generated vs Scraped Content

- **Theory** → Scraped from official docs (authoritative)
- **ELI5** → LLM-generated (creative, requires review via confidence scores)
- **Examples** → Mix of scraped (from docs) and LLM-generated (more variety)
- **Exercises** → LLM-generated (needs solution verification)

### 10.3 Single User vs Multi User (MVP)

- **MVP:** Single user, progress in localStorage (React side)
- **Future:** User accounts, progress tracking, spaced repetition

### 10.4 Cost

- **$0 total** — Ollama runs locally, all docs are public, PostgreSQL + Weaviate in Docker
- No cloud services, no API keys, no subscriptions
